# Requirements Document

## Introduction

This feature implements property-based tests (PBT) for the 5 most critical systemic invariants
identified in the EscapeKit / CodeMemória codebase. The tests use **fast-check** (TypeScript/JavaScript)
and **Hypothesis** (Python) to generate hundreds of randomised inputs and verify that each invariant
holds unconditionally. The goal is to catch regressions that example-based tests miss and to serve
as executable, machine-checked documentation of the system's core guarantees.

## Glossary

- **PBT**: Property-Based Test — a test that asserts a property holds for all inputs generated by an
  arbitrary/strategy, rather than for a fixed set of examples.
- **Arbitrary**: A fast-check generator that produces random values of a given shape.
- **Strategy**: A Hypothesis generator (equivalent to fast-check Arbitrary in Python).
- **Invariant**: A property of the system that must hold for every valid input, regardless of order
  or magnitude.
- **computeRiskLevel**: Pure exported function in `src/governance/GovernanceEngine.ts` that maps
  issues and compliance stamps to a `RiskLevel`.
- **RiskLevel**: One of the string literals `'low' | 'medium' | 'high' | 'critical'`.
- **FederatedServer**: The FastAPI server in `federated-server/main.py`.
- **success_rate**: The field `success_rate` in `FederatedPatternResponse`, computed as
  `min(1.0, success_count / max(1, total_pushes))`.
- **confidence**: The field `confidence` in `FederatedPatternResponse`, computed as
  `max(0.0, min(1.0, cosine_similarity))`.
- **CreditTracker**: The class in
  `pisosrealview-pro-transformed/backend/services/gateway/CreditTracker.js` that tracks monthly
  API credit usage per provider.
- **_ensureMonth**: The private method of `CreditTracker` that resets counters when the calendar
  month changes.
- **CircuitBreaker**: The class in `src/utils/circuitBreaker.ts` that implements the circuit-breaker
  pattern with states `CLOSED`, `OPEN`, and `HALF_OPEN`.
- **computeFingerprint**: Pure exported function in `src/governance/utils/fingerprint.ts` that
  produces a `CodeFingerprint` from a source-code string.
- **computeSimilarity**: Pure exported function in `src/governance/utils/fingerprint.ts` that
  returns a similarity score in `[0, 1]` between two `CodeFingerprint` objects.
- **CodeFingerprint**: Object with fields `hash: string`, `astSignature: string`,
  `dependencies: string[]`, `complexity: number`.
- **_InMemoryStore**: The in-memory `_PatternStore` implementation in `federated-server/main.py`
  used to replace the Chroma store during tests.

## Requirements

### Requirement 1: computeRiskLevel — Determinism and Correctness

**User Story:** As a governance engineer, I want property-based tests for `computeRiskLevel`, so
that I can be confident the risk classification logic is deterministic and correct for all possible
combinations of issues and compliance scores.

#### Acceptance Criteria

1. THE PBT_Suite SHALL generate at least 1000 random `(issues, complianceStamps)` pairs and verify
   that `computeRiskLevel` returns one of `'low' | 'medium' | 'high' | 'critical'` for every pair.

2. WHEN any issue in the input array has `severity === 'error'`, THE PBT_Suite SHALL verify that
   `computeRiskLevel` returns `'critical'`.

3. WHEN no issue has `severity === 'error'` AND at least one issue has `severity === 'warning'`
   AND the average `score` across all `complianceStamps` is less than `0.7`, THE PBT_Suite SHALL
   verify that `computeRiskLevel` returns `'high'`.

4. WHEN no issue has `severity === 'error'` AND at least one issue has `severity === 'warning'`
   AND the average `score` across all `complianceStamps` is greater than or equal to `0.7`,
   THE PBT_Suite SHALL verify that `computeRiskLevel` returns `'medium'`.

5. WHEN no issue has `severity === 'error'` AND no issue has `severity === 'warning'`,
   THE PBT_Suite SHALL verify that `computeRiskLevel` returns `'low'`.

6. THE PBT_Suite SHALL verify that calling `computeRiskLevel` twice with identical inputs returns
   the same value (determinism).

7. WHEN `complianceStamps` is an empty array AND no issue has `severity === 'error'` AND at least
   one issue has `severity === 'warning'`, THE PBT_Suite SHALL verify that `computeRiskLevel`
   returns `'medium'` (empty stamps default to avgScore `1.0`).

### Requirement 2: FederatedServer — success_rate and confidence Bounds

**User Story:** As a backend engineer, I want property-based tests for the `GET /query` endpoint,
so that I can guarantee `success_rate` and `confidence` are always in `[0.0, 1.0]` regardless of
the stored data.

#### Acceptance Criteria

1. THE PBT_Suite SHALL generate at least 200 random `(success_count, total_pushes)` pairs with
   `success_count >= 1` and `total_pushes >= 1` and verify that the computed `success_rate` is
   always in `[0.0, 1.0]`.

2. WHEN `success_count <= total_pushes`, THE PBT_Suite SHALL verify that `success_rate` equals
   `success_count / total_pushes` exactly (no capping needed).

3. WHEN `success_count > total_pushes`, THE PBT_Suite SHALL verify that `success_rate` equals
   `1.0` (capped).

4. THE PBT_Suite SHALL generate at least 200 random 384-dimensional float vectors and verify that
   the `confidence` field returned by `GET /query` is always in `[0.0, 1.0]`.

5. WHEN the query vector is the zero vector, THE PBT_Suite SHALL verify that `confidence` equals
   `0.0` (cosine similarity with zero vector is defined as `0.0`).

6. THE PBT_Suite SHALL use `_InMemoryStore` to replace the Chroma store so tests run without
   external dependencies.

### Requirement 3: CreditTracker — Idempotency and Isolation

**User Story:** As a gateway engineer, I want property-based tests for `CreditTracker`, so that I
can guarantee monthly rollover is idempotent, counters never go negative, and `increment` is
correctly reflected in `getState`.

#### Acceptance Criteria

1. THE PBT_Suite SHALL generate at least 1000 random call counts `N >= 1` and verify that calling
   `_ensureMonth()` N times within the same calendar month produces the same `state` as calling it
   once (idempotency).

2. WHEN the injected clock advances past a month boundary, THE PBT_Suite SHALL verify that
   `_ensureMonth()` resets `state.counters` to `{}` and updates `state.month` to the new month.

3. THE PBT_Suite SHALL generate random `(providerId, limit, usedCount)` triples and verify that
   `getState(providerId, limit).remaining` equals `Math.max(0, limit - used)` for any `limit > 0`.

4. THE PBT_Suite SHALL verify that `getState(providerId, 0).remaining` is always `null`.

5. THE PBT_Suite SHALL generate random `(providerId, initialUsed)` pairs and verify that after
   calling `increment(providerId)`, `getState(providerId, limit).used` equals `initialUsed + 1`.

6. THE PBT_Suite SHALL use the injected `clock` option to control time without touching the
   filesystem clock, and SHALL use an in-memory `fsModule` stub to avoid real file I/O.

### Requirement 4: CircuitBreaker — State Transitions and Monotonicity

**User Story:** As a reliability engineer, I want property-based tests for `CircuitBreaker`, so
that I can guarantee state transitions are correct and counters never violate their invariants.

#### Acceptance Criteria

1. THE PBT_Suite SHALL generate at least 1000 random sequences of `execute` calls (mix of
   successes and failures) and verify that `getStats().state` is always one of
   `'CLOSED' | 'OPEN' | 'HALF_OPEN'`.

2. THE PBT_Suite SHALL verify that `getStats().failureCount` is always `>= 0` after any sequence
   of operations.

3. THE PBT_Suite SHALL verify that `getStats().successCount` is always `>= 0` after any sequence
   of operations.

4. WHEN `failureThreshold` consecutive failures occur from `CLOSED` state, THE PBT_Suite SHALL
   verify that the state transitions to `OPEN`.

5. WHEN a success occurs in `HALF_OPEN` state, THE PBT_Suite SHALL verify that the state
   transitions to `CLOSED` and `failureCount` resets to `0`.

6. WHEN a failure occurs in `HALF_OPEN` state, THE PBT_Suite SHALL verify that the state
   transitions back to `OPEN`.

7. WHEN the circuit is `OPEN` and the injected clock has advanced by at least `recoveryTimeoutMs`
   milliseconds since the last failure, THE PBT_Suite SHALL verify that `canExecute()` returns
   `true`.

8. WHEN `reset()` is called, THE PBT_Suite SHALL verify that the state is `'CLOSED'` and
   `failureCount` is `0`.

### Requirement 5: computeFingerprint and computeSimilarity — Reflexivity and Bounds

**User Story:** As a governance engineer, I want property-based tests for `computeFingerprint` and
`computeSimilarity`, so that I can guarantee the fingerprint is deterministic and similarity scores
are always valid.

#### Acceptance Criteria

1. THE PBT_Suite SHALL generate at least 1000 random source-code strings and verify that
   `computeFingerprint(code).complexity >= 0` for every input.

2. THE PBT_Suite SHALL verify that `computeFingerprint(code).hash` is deterministic: calling
   `computeFingerprint` twice with the same string always returns the same `hash`.

3. THE PBT_Suite SHALL generate at least 1000 random `CodeFingerprint` pairs and verify that
   `computeSimilarity(a, b)` is always in `[0.0, 1.0]`.

4. THE PBT_Suite SHALL verify that `computeSimilarity(fp, fp) === 1.0` for any fingerprint
   (reflexivity).

5. THE PBT_Suite SHALL verify whether `computeSimilarity(a, b) === computeSimilarity(b, a)`
   (symmetry) holds; IF symmetry does not hold for any generated pair, THE PBT_Suite SHALL record
   the counterexample and document the asymmetry as a known limitation.

6. WHEN both fingerprints have empty `dependencies` arrays, THE PBT_Suite SHALL verify that the
   Jaccard component contributes `0.0` and the result is still in `[0.0, 1.0]`.

7. WHEN both fingerprints have `complexity === 0`, THE PBT_Suite SHALL verify that the complexity
   component contributes `1.0` (no difference) and the result is still in `[0.0, 1.0]`.
